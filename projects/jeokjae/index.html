02

Redis 없는 asyncio WebSocket 파이프라인 (6채널)

왜 자체 구현인가
  • Django Channels는 Redis Pub/Sub 브로커가 사실상 필수 → 6주 일정에 운영 복잡도 과다
  • 프로세스 내 asyncio.Queue 기반 pub/sub로 외부 브로커 0개로 6채널 실시간 브로드캐스트
  • 동기 HTTP 뷰 → ASGI 루프로 call_soon_threadsafe 안전 전달. 한계(멀티워커 시 신호 비공유)는 ADR로 명시
핵심 코드 (config/pubsub.py)
def notify(channel: str) -> None:
    """DB 업데이트 후 sync thread에서 안전 호출 — 모든 WS 구독자에 즉시 신호."""
    if _loop is None or not _loop.is_running():
        return
    with _lock:
        queues = list(_subscribers.get(channel, set()))
    if not queues:
        return

    def _put() -> None:
        for q in queues:
            try:
                q.put_nowait(True)
            except asyncio.QueueFull:
                pass  # 처리 중이면 스킵 (다음 notify가 커버)

    _loop.call_soon_threadsafe(_put)

WS 핸들러는 접속 즉시 최신 snapshot을 보내고, 이후 notify 수신 시 즉시 재전송 · 신호가 없어도 10초 폴링 refresh · 5초 heartbeat로 연결을 유지합니다.

실시간 / 중복

blocked 안전 이벤트가 매 tick 중복 생성되던 문제

문제

Isaac Sim이 1초마다 배치를 보내는데, AMR이 정체(blocked) 상태를 유지하면 매 tick마다 SafetyEvent가 새로 생성돼 동일 상황에 수백 건이 쌓였습니다.

원인

상태 감지 로직이 '현재 blocked인가'만 보고 '이미 이 정체 세션에서 이벤트를 만들었는가'를 추적하지 않았습니다. 매 요청 DB 조회로 막으면 비용이 큽니다.

해결

정체 세션 식별자(status_changed_at)를 키로 하는 인메모리 캐시로 세션당 1회만 생성. 캐시 미스(서버 재시작 등)일 때만 DB 미확인 이벤트 존재를 1회 확인해 캐시를 복구합니다.

배운 점

실시간 시스템은 '상태 저장'이 아니라 '상태 전환 감지 + 중복 방지'가 핵심임을 체감. 같은 blocked 세션은 정확히 1건만 기록되도록 정리했습니다.

✕ Before
# 매 tick blocked면 그대로 생성 → 수백 건
if snapshot.status == "blocked":
    SafetyEvent.objects.create(
        event_type="obstacle_blocked",
        source_id=snapshot.amr_id, ...
    )
✓ After
# 세션(status_changed_at) 단위 인메모리 캐시
if _blocked_event_created.get(amr_id) == snapshot.status_changed_at:
    return False                       # 같은 세션 → 스킵 (DB 조회 X)

if SafetyEvent.objects.filter(         # 캐시 미스 시 1회만 확인
        event_type="obstacle_blocked",
        source_id=amr_id, acknowledged=False).exists():
    _blocked_event_created[amr_id] = snapshot.status_changed_at
    return False

SafetyEvent.objects.create(event_type="obstacle_blocked", ...)
_blocked_event_created[amr_id] = snapshot.status_changed_at
동시성

다중 시뮬레이터 폴링 시 같은 명령이 중복 실행되던 문제

문제

DeviceCommand 큐를 Isaac Sim이 /control/pending/으로 폴링하는 구조에서, 여러 인스턴스가 동시에 폴링하면 같은 PENDING 명령을 둘 다 받아 2번 실행하는 race condition이 발생했습니다.

원인

조회와 status 갱신 사이가 비원자적이라, 두 폴링이 같은 행을 동시에 가져갔습니다.

해결

transaction.atomic() + select_for_update()로 pending 행을 잠그고, 한 번에 status='sent'로 일괄 전환했습니다. 조회는 .values()로 필요한 컬럼만 가져옵니다.

배운 점

DB를 큐로 쓸 때 '꺼내가는' 연산은 반드시 행 잠금 + 원자적 상태 전환으로 묶어야 한다는 걸 배웠습니다.

배포 / 침묵형 실패

fixture가 없으면 batch-update가 조용히 0건 처리되던 문제

문제

새 환경에 배포하면 Isaac Sim → Django batch-update는 200 OK인데 AMR 카드가 끝까지 비어 있었습니다.

원인

초기에 쓰던 filter().update() 방식은 DB에 해당 AMR 레코드가 없으면 매칭 0건으로 'silent fail'. 응답이 200이라 로그만으론 정상으로 보였습니다.

해결

신규 장비는 update_or_create로 자동 생성되도록 처리하고, 환경 세팅 시 fixture를 선 로드하도록 RUN_AND_TEST_GUIDE.md / TROUBLESHOOTING.md로 가이드화했습니다.

배운 점

200 OK가 곧 '정상 동작'은 아니라는 것. 신규 팀원 환경 세팅 시간도 함께 단축했습니다.

프론트 / 복원력

WebSocket 단절 시 대시보드 화면이 고착되던 문제

문제

WS 연결이 끊기면 마지막 수신 상태가 그대로 표시돼, 운영자가 실제 장비 상태 변화를 모르는 상황이 생겼습니다.

원인

WS 실패 시 폴링으로 전환하는 로직이 없어, 연결만 시도하고 실패하면 이후 업데이트가 멈췄습니다.

해결

각 Pinia 스토어에 WS → 15초 REST 폴링 → 캐시 유지(degraded 배지) 3단 폴백을 구현했습니다. 화면 데이터 공백을 최대 15초 이내로 제한했습니다.

배운 점

네트워크가 불안정해도 '죽지 않는 UI'를 만드는 것이 실무 UX에서 더 중요하다는 걸 배웠습니다.
