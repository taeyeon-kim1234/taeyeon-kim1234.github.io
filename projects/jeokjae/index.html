F.

트러블슈팅

"정상 응답인데 데이터가 비어 있는" 침묵형 실패와, 다중 폴링 환경의 동시성 문제를 다뤘습니다.

fixture가 없으면 batch-update가 조용히 0건 처리되던 문제

문제

Isaac Sim → Django batch-update API로 AMR 상태를 10초마다 전송하는데, 새 환경에 배포하면 AMR 카드가 끝까지 비어 있었습니다.

원인

Django ORM의 filter().update() 방식 한계 — DB에 해당 AMR 레코드가 없으면 매칭 0건으로 'silent fail'. 응답이 200 OK라 로그만 봐서는 정상으로 보였습니다.

해결

amr_bootstrap 등 fixture 6개를 환경 세팅 시 선 로드하도록 가이드. TROUBLESHOOTING.md / WEB_CONCEPTS_GUIDE.md / RUN_AND_TEST_GUIDE.md 작성.

배운 점

신규 팀원 환경 세팅 시간을 단축했고, 향후 update_or_create 또는 PostgreSQL upsert로 근본 개선하는 ADR을 기록했습니다. (관련 MR · !124 ROS 토픽 웹 연동, 멘토 리뷰)

다중 시뮬레이터 폴링 시 동일 명령이 중복 전달되던 문제

문제

웹의 '명령 발행' → DeviceCommand 큐 적재 → Isaac Sim이 /control/pending/ 폴링 구조에서, 여러 시뮬레이터가 동시에 폴링하면 같은 명령을 2번 받아 실행하는 race condition이 발생했습니다.

원인

조회와 상태 갱신 사이가 비원자적이라, 두 폴링이 같은 PENDING 행을 동시에 가져갔습니다.

Before
pending = DeviceCommand.objects \
    .filter(status='PENDING')
for cmd in pending:
    cmd.status = 'TAKEN'
    cmd.save()
해결 — 원자성 + 행 잠금

DRF view에서 transaction.atomic() + select_for_update()로 행을 잠갔습니다. 조회 단계에서 .only() / .iterator()로 메모리·쿼리 부담도 함께 정리했습니다. 

with transaction.atomic():
    qs = DeviceCommand.objects.select_for_update()
    pending = qs.filter(status='PENDING')[:N]
부수적으로 정리한 것

CONN_MAX_AGE DB 커넥션 재사용, fetchWithTimeout() 프론트 타임아웃, 제어/취소 요청 CSRF 헤더, Caddy vs Nginx 선택 근거 ADR 문서화. (관련 MR · !215 룰엔진+명령 큐 / !217 제어 폴링 안정화)

in the end

모델보다 환경과 좌표를 먼저 의심한다

강화학습 모델이 잘 학습되는지보다, 학습 환경과 평가 좌표가 정확한지를 먼저 의심해야 한다는 것을 배웠습니다. 추정 좌표로 8차까지 학습한 뒤 실측에서 3.8m 오차를 발견한 시점, 가설을 의심하고 좌표 데이터 자체를 갈아엎는 결정을 내렸습니다. 학습 일지로 의사결정 근거를 남기는 습관이 이때 자리잡았습니다. RL 단독 실험에서 시작해 Django/Vue 통합, ADR·문서 작성까지 책임이 확장되며 사실상 코어 멤버 역할을 수행했습니다.
