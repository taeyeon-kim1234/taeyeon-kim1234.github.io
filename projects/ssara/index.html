PROJECT 02 · 공통 프로젝트

자율주행 로봇 '싸라'가 노인의 말을 듣고, 주변을 인식하고, 보호자에게 알리는 스마트 케어 솔루션.
AI 음성비서·YOLOv8 커스텀 모델·VSLAM 좌표 브릿지를 ROS 2 노드로 직접 구현했습니다.

기획 배경

고령 인구 증가와 독거 노인의 안전 사각지대를 줄이기 위해, 자율주행 로봇이 음성 명령을 받고·실내를 인식하며·보호자와 연결되는 케어 플랫폼을 기획했습니다. 저는 로봇의 '귀와 입'(AI 음성비서)과 '눈'(YOLOv8 객체탐지), '위치 감각'(VSLAM 브릿지) 세 영역을 맡았습니다.

내가 사용한 기술과 선정 이유

ROS 2 노드 하나가 STT → 명령 파싱 → 로봇 제어 / 웹 전송까지 이어지는 파이프라인을 담당합니다. 응답 속도·오프라인 안정성·팀원과의 데이터 계약을 기준으로 기술을 선택했습니다.

faster-whisper (STT)

base → tiny 교체로 추론 속도 ~3배 향상. initial_prompt로 한국어 발음 변형 힌트를 제공해 인식률 보정.

gTTS + espeak fallback

자주 쓰는 응답은 시작 시 MP3 캐시 생성. 네트워크 오류 시 espeak 오프라인 fallback으로 TTS 안정화.

MQTT (paho)

robot/command·robot/pose·guardian 토픽으로 ROS 2 ↔ 웹 양방향 연결. JSON 프로토콜 직접 설계.

RTAB-Map odom 브릿지

BEST_EFFORT QoS 매칭, (0,0) 리셋·좌표 점프·떨림 4단계 필터링으로 SLAM 위치 노이즈 억제.

YOLOv8 (Ultralytics)

실내 환경의 신호등·정지선·장애물 데이터셋 직접 구성. BoxF1·PR 커브 분석으로 v1→v7 반복 고도화.

ROS 2 launch 분리

voice_map / voice_fast / voice_final 3개 launch 파일로 개발·최적화·배포 모드를 단계별로 분리. setup.py entry point 등록.

담당 기능 AI / Vision

AI 음성비서 파이프라인 전체 구현

마이크 입력 → faster-whisper STT → fuzzy matching 명령 분류 → 멈춰 / 앞으로 / 병원·약국 검색 / 보호자 호출 / 현재 시간 / 거리 안내 / YOLO 신호등 반응 등 10+ 명령을 처리하는 ROS 2 노드를 구현했습니다. 인식 시작·종료를 beep 음으로 피드백하고, 자주 쓰는 응답은 시작 시 MP3로 미리 캐시합니다.

04

YOLOv8 커스텀 모델 학습 (v1–v7)

실내 환경의 신호등(빨간불/초록불) · 정지선 · 장애물 인식을 위해 데이터셋을 직접 구성하고 v1 → v7 총 7회 반복 학습. 각 버전마다 BoxF1 · PR 커브 · confusion matrix를 분석해 취약 클래스 데이터를 추가 수집하고 하이퍼파라미터를 조정했습니다. 최종 모델(best.pt v7)은 stt_module_fast의 YOLO 콜백에 연결해 "빨간불 → 정지 / 초록불 → 출발" 자동 반응을 구현했습니다.

best.pt (v7)dataset/ (커스텀 라벨)stt_module_fast.py (YOLO callback)

트러블슈팅

STT 지연·MQTT 연동·SLAM QoS·YOLO 인식률까지, 실시간 파이프라인에서 부딪힌 문제들과 해결.

STT 응답 지연 — base 모델 2~3초

문제

faster-whisper base 모델로 STT를 처리했을 때 명령 한 마디에 2~3초 지연이 발생. 로봇이 명령을 늦게 실행하거나, 사용자가 같은 말을 반복하는 문제가 생겼습니다.

원인

base 모델의 파라미터 수(74M)가 실시간 명령 처리에 과했습니다. 간단한 명령어 인식에는 tiny(39M)로도 충분하지만, 한국어 발음 변형에 취약할 수 있었습니다.

해결 
# base → tiny 교체 + initial_prompt로 발음 힌트 제공
self.model = WhisperModel("tiny", device="cpu", compute_type="int8")

# 발음 변형 힌트 (싸라·병원·멈춰 등 핵심 단어)
context_words = "싸라, 사라, 병원, 약국, 멈춰, 앞으로, 보호자..."
segments, _ = self.model.transcribe(wav_buffer, language="ko",
                                     initial_prompt=context_words)
결과

응답 속도 ~3배 향상. 발음 변형은 initial_prompt + fuzzy matching 조합으로 보완해 인식률 하락 없이 속도를 개선했습니다.

웹에 JSON 기계어 노출

문제

초기에 MQTT 메시지를 그대로 웹으로 전송했더니 보호자 앱 화면에 JSON 키값이나 명령 코드가 그대로 표시됐습니다.

원인

ROS 2 노드 내부 명령 코드(navigation, guardian_alert 등)와 UI 표시 문자열이 분리되지 않은 채로 전송됐습니다.

해결 
# type(내부 코드)과 display_text(UI 표시용)를 분리
payload = {
    "type": "guardian_alert",
    "display_text": "보호자에게 메시지를 보냈습니다.",
    "location": {x: pos_x, y: pos_y},
    "timestamp": now.isoformat()
}
self.mqtt_client.publish("guardian", json.dumps(payload))
결과

프론트엔드는 display_text만 렌더링하면 되는 계약이 명확해졌고, 팀 내 MQTT 메시지 포맷을 표준화하는 계기가 됐습니다.

RTAB-Map odom QoS 불일치로 위치 수신 불가

문제

pose_bridge 노드가 /rtabmap/odom을 구독했지만 메시지를 전혀 수신하지 못했습니다. 토픽 자체는 존재했고, 다른 노드에서는 같은 토픽이 정상 동작했습니다.

원인

RTAB-Map은 /rtabmap/odom을 BEST_EFFORT로 퍼블리시하는데, 구독자의 QoS가 기본값 RELIABLE로 설정되어 있어 ROS 2 QoS 협상이 실패했습니다.

해결 
from rclpy.qos import QoSProfile, ReliabilityPolicy

# 퍼블리셔(RTAB-Map)와 동일하게 BEST_EFFORT로 맞춤
qos = QoSProfile(depth=10, reliability=ReliabilityPolicy.BEST_EFFORT)
self.subscription = self.create_subscription(
    Odometry, '/rtabmap/odom', self.listener_callback, qos
)
결과

위치 수신 즉시 정상화. 이후 수신된 좌표에 4단계 필터를 추가해 실내 SLAM 노이즈(좌표 점프·떨림)도 처리했습니다.

YOLOv8 v1~v4 낮은 인식률

문제

1차 데이터셋으로 학습한 v1 모델은 실내 조명과 배경에 따라 신호등·정지선 인식률이 기대에 크게 못 미쳤습니다. 특히 일부 클래스(정지선)는 Recall이 매우 낮았습니다.

원인

초기 데이터셋의 클래스 불균형과 실내 조명·각도 다양성 부족. 데이터 수량보다 환경 다양성이 더 중요한 문제였습니다.

해결

각 버전(v1~v7)마다 BoxF1 곡선 · PR 커브 · confusion matrix를 분석해 취약 클래스의 데이터를 우선 추가 수집하고, augmentation(밝기·회전·크롭)을 강화. v4부터 안정적인 인식률 확보, v7에서 목표치 달성 후 stt_module_fast YOLO 콜백에 연결.

결과

단순히 epoch를 늘리는 것보다 데이터 품질·클래스 균형이 성능에 더 크게 영향한다는 걸 직접 확인했습니다. 이후 AI 파이프라인 디버깅 시 먼저 confusion matrix를 보는 습관이 생겼습니다.

retrospective

파이프라인을 단계별로 쪼개는 것의 힘

가장 크게 배운 점은 AI 파이프라인은 모델 성능보다 "어느 단계에서 무엇이 실패하는지"를 빠르게 파악하는 구조가 더 중요하다는 것이었습니다. STT 지연인지, 명령 파싱 오류인지, MQTT 전송 문제인지 분리해서 볼 수 있게 launch 파일과 노드를 단계별로 나눠놓은 것이 디버깅 속도를 크게 줄였습니다. 다음엔 fuzzy matching 대신 경량 분류 모델을 붙여 명령 오인식을 줄여보고 싶습니다.
