STT 경량화

전체를 한 노드로 합치자 — 네트워크 이슈로 음성 인식이 버벅거렸다

문제

초기엔 STT·YOLO·네비게이션·MQTT 전송을 하나의 ROS 2 노드에 통합해 실행했습니다. 이 상태에서 Wi-Fi나 네트워크 부하가 조금만 높아져도 Whisper 추론이 지연되고, 응답 시간이 3~5초까지 늘어났습니다.

원인

Whisper base(74M 파라미터)는 CPU에서 이미 무거운데, YOLO 추론·네트워크 I/O·MQTT publish가 같은 프로세스에서 경합하면서 GIL 병목과 리소스 충돌이 발생했습니다.

해결

노드를 역할별로 분리(stt_module_map → stt_module_fast)하고, Whisper를 tiny(39M)로 교체해 추론 단계만 ~3배 빠르게 만들었습니다. 자주 쓰는 응답은 시작 시 MP3로 미리 캐시해 TTS 지연도 제거했습니다.

배운 점

AI 파이프라인에서 '무거움'의 원인을 모델 크기·I/O 경합·GIL 중 어디서 오는지 분리하는 게 먼저라는 걸 배웠습니다. 모델을 바꾸기 전에 병목 위치를 먼저 확인해야 했습니다.

Whisper

Whisper가 명령어를 엉뚱하게 인식 — 한국어 발음 변형

문제

'싸라'를 '사라·자라·살아'로, '병원'을 '평원·캬원'으로 인식하는 경우가 자주 발생했습니다. tiny 모델로 교체 후에는 특히 짧은 명령어에서 오인식이 더 잦아졌습니다.

원인

Whisper는 일반 회화 데이터로 학습되어 로봇 명령어처럼 짧고 특정 단어 중심의 발화에 약합니다. 특히 Korean ASR에서 받침이 있는 단어는 유사 발음으로 대체되는 경향이 있습니다.

해결

initial_prompt에 핵심 명령어와 발음 변형 후보를 힌트로 주입하고, 인식 결과에 fuzzy matching(difflib SequenceMatcher)을 적용해 유사도 0.6 이상이면 명령으로 처리했습니다. 또한 Whisper 환각(hallucination) 패턴 단어 목록을 만들어 필터링했습니다.

배운 점

STT 모델의 한계를 모델 교체로만 해결하려 하면 끝이 없습니다. 입력 단계(힌트) + 출력 단계(후처리)를 함께 다루는 게 더 효율적이었습니다.

✕ Before
# 인식 결과를 그대로 명령어와 비교
text = self.transcribe(audio)
if "멈춰" in text:
    self.stop_robot()
elif "병원" in text:
    self.navigate("hospital")
# '평원', '뱅원' 등 오인식 시 명령 미실행
✓ After
# initial_prompt + fuzzy matching 조합
CONTEXT = "싸라, 사라, 병원, 약국, 멈춰, 멈춰라, 앞으로, 보호자"
HALLUCINATIONS = ["감사합니다", "수고하세요", "안녕하세요"]

segments, _ = self.model.transcribe(wav, language="ko",
                                     initial_prompt=CONTEXT)
text = " ".join(s.text for s in segments).strip()

# 환각 필터
if any(h in text for h in HALLUCINATIONS): return

# fuzzy matching으로 명령 분류
best, score = max(COMMANDS.items(),
    key=lambda kv: SequenceMatcher(None, text, kv[0]).ratio())
if score >= 0.6:
    self.execute(best)
Whisper

주변 소음·짧은 침묵에 Whisper가 무한 대기하거나 환각 생성

문제

조용한 환경에서 아무 말 없이 마이크가 켜지면 Whisper가 없는 말을 만들어내거나(환각), 시끄러운 환경에서는 소음을 발화로 인식해 의도치 않은 명령이 실행됐습니다.

원인

SpeechRecognition의 에너지 임계값이 고정값이면 환경에 따라 소음을 말로, 말을 침묵으로 잘못 판단합니다. Whisper 자체도 짧은 오디오에서 환각 텍스트를 생성하는 특성이 있습니다.

해결

dynamic_energy_threshold=True로 주변 소음에 자동 적응하고, adjust_for_ambient_noise()로 매 발화 전 보정했습니다. Whisper 출력이 일정 길이 미만이거나 환각 패턴이면 무시하는 후처리도 추가했습니다.

배운 점

실제 사용 환경(실내 로봇 옆)은 조용한 녹음 환경과 전혀 다릅니다. 모델 성능보다 입력 품질 보장이 더 중요했습니다.

✕ Before
self.recognizer.energy_threshold = 300  # 고정값
self.recognizer.dynamic_energy_threshold = False
# 소음 환경에서 오작동, 조용한 환경에서 환각
✓ After
# 환경 적응형 설정
self.recognizer.energy_threshold = 300
self.recognizer.dynamic_energy_threshold = True
self.recognizer.dynamic_energy_adjustment_damping = 0.15
self.recognizer.dynamic_energy_ratio = 1.5
self.recognizer.pause_threshold = 0.8

with mic as source:
    self.recognizer.adjust_for_ambient_noise(source, duration=0.5)
    audio = self.recognizer.listen(source, timeout=5,
                                   phrase_time_limit=8)
MQTT

Wi-Fi 변경 시 MQTT 연결이 끊기고 음성 명령이 웹에 전달되지 않음

문제

로봇이 장소를 이동하거나 AP가 바뀌면 MQTT 브로커와의 연결이 끊겼고, 이후 음성 명령을 내려도 웹에 아무 반응이 없었습니다. 연결이 끊긴 줄 모르고 계속 명령을 반복하는 상황이 생겼습니다.

원인

paho-mqtt 클라이언트를 단순 connect()만으로 초기화하면, 네트워크가 변경됐을 때 자동으로 재연결을 시도하지 않습니다. 브로커 주소가 하드코딩된 localhost여서 IP가 바뀌면 연결 자체가 불가능했습니다.

해결

reconnect_delay_set()으로 자동 재연결 정책을 설정하고, on_disconnect 콜백에서 재연결을 명시적으로 트리거했습니다. 브로커 주소를 환경변수로 분리해 네트워크 변경 시 재시작 없이 대응 가능하게 했습니다.

배운 점

IoT 환경에서 네트워크는 언제든 끊길 수 있습니다. 연결 자체보다 재연결 정책이 더 중요하고, 연결 상태를 사용자에게 피드백해야 한다는 걸 배웠습니다.

✕ Before
self.client = mqtt.Client(...)
self.client.connect("localhost", 1883, 60)
# Wi-Fi 변경 시 연결 끊김 → 재연결 없음
# 명령 전송 실패해도 알 수 없음
✓ After
self.client = mqtt.Client(...)
self.client.reconnect_delay_set(min_delay=1, max_delay=10)

def on_disconnect(client, userdata, rc):
    if rc != 0:
        print("⚠️ MQTT 연결 끊김, 재연결 시도...")
        try:
            client.reconnect()
        except Exception:
            time.sleep(3)
            client.reconnect()

self.client.on_disconnect = on_disconnect
broker = os.getenv("MQTT_BROKER", "localhost")
self.client.connect(broker, 1883, 60)
self.client.loop_start()  # 백그라운드 재연결 루프
SLAM

RTAB-Map odom QoS 불일치로 위치 수신 불가

문제

pose_bridge 노드가 /rtabmap/odom을 구독했지만 메시지를 전혀 수신하지 못했습니다. 토픽은 존재했고 다른 노드에서도 같은 토픽이 정상 동작했습니다.

원인

RTAB-Map은 /rtabmap/odom을 BEST_EFFORT로 퍼블리시하는데, 구독자의 QoS가 기본값 RELIABLE로 설정되어 ROS 2 QoS 협상이 실패했습니다.

해결

QoSProfile을 BEST_EFFORT로 맞추고, 수신된 좌표에 (0,0) 리셋 감지 / 좌표 점프 필터 / 떨림 억제 4단계를 추가했습니다.

배운 점

ROS 2에서 토픽이 존재해도 QoS가 맞지 않으면 메시지를 받을 수 없습니다. ros2 topic info --verbose로 QoS를 먼저 확인하는 습관이 생겼습니다.

✕ Before
# 기본 QoS (RELIABLE) — RTAB-Map과 불일치
self.subscription = self.create_subscription(
    Odometry,
    '/rtabmap/odom',
    self.listener_callback,
    10  # 기본값: RELIABLE → 메시지 수신 안 됨
)
✓ After
from rclpy.qos import QoSProfile, ReliabilityPolicy

# RTAB-Map 퍼블리셔와 동일하게 BEST_EFFORT
qos = QoSProfile(depth=10,
                 reliability=ReliabilityPolicy.BEST_EFFORT)
self.subscription = self.create_subscription(
    Odometry,
    '/rtabmap/odom',
    self.listener_callback,
    qos
)
YOLO

YOLOv8 v1~v4 — 정지선 Recall 극히 낮음

문제

v1~v4 모델은 빨간불·초록불은 어느 정도 인식했지만, 정지선 클래스의 Recall이 0.2 이하로 실질적으로 작동하지 않았습니다.

원인

초기 데이터셋에서 정지선 이미지가 신호등 대비 절대적으로 적었고, 촬영 각도·조명 다양성도 부족했습니다. 클래스 불균형이 심하면 모델이 다수 클래스에 편향됩니다.

해결

각 버전(v1~v7)마다 BoxF1 · PR 커브 · confusion matrix를 분석해 취약 클래스 데이터를 우선 추가 수집하고, augmentation(밝기·회전·크롭)을 강화. v7에서 목표 인식률 달성 후 stt_module_fast YOLO 콜백에 연결했습니다.

배운 점

epoch를 늘리는 것보다 클래스 균형과 데이터 다양성이 성능에 훨씬 크게 영향합니다. confusion matrix로 어느 클래스가 취약한지 먼저 파악하는 게 가장 빠른 개선 경로였습니다.

retrospective

파이프라인을 단계별로 쪼개는 것의 힘

가장 크게 배운 점은 AI 파이프라인은 모델 성능보다 "어느 단계에서 무엇이 실패하는지"를 빠르게 파악하는 구조가 더 중요하다는 것이었습니다. STT 지연인지, 명령 파싱 오류인지, MQTT 전송 문제인지 분리해서 볼 수 있게 launch 파일과 노드를 단계별로 나눠놓은 것이 디버깅 속도를 크게 줄였습니다. 다음엔 fuzzy matching 대신 경량 분류 모델을 붙여 명령 오인식을 줄여보고 싶습니다.
