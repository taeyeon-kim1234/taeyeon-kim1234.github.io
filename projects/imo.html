인증 토큰 갱신 중복 호출

문제

홈 진입처럼 여러 API가 동시에 호출되는 상황에서 access token이 만료되면 여러 요청이 동시에 401을 받아 각자 refresh를 호출했고, refresh token rotation 환경에서 둘째 요청부터 invalid가 발생할 위험이 있었습니다.

해결

pendingRefresh Future를 공유하는 single-flight 패턴으로 refresh를 1회로 묶고, refresh 실패 원인을 unauthorized / network / server / unknown 4종으로 분리해 401·403일 때만 logout하도록 정책 조정.

50Hz 실시간 표시 데이터로 인한 HUD 재빌드

문제

glass_display_data가 약 50Hz로 들어오는데, 수신마다 notifyListeners()를 호출하면 HUD가 불필요하게 재빌드. EMG 1~4는 이 메시지에만 실려 다른 메시지를 받을 때 채널이 비는 문제도 있었습니다.

해결

80ms throttled notify로 입력 50Hz → 약 12.5Hz UI notify로 제한 (~75% 감소). 상태 이벤트는 즉시 반영하되 표시 데이터만 throttle. EMG 채널은 마지막 값 carry-over로 메시지 타입 차이에도 게이지가 사라지지 않게.

캘리브레이션 미응답과 Pi race condition

문제

sensors_attached 직후 start_calibration을 바로 보내면 Pi 상태머신이 phase 전환을 끝내기 전에 다음 명령을 받아 collection이 시작되지 않았고, 응답 미수신 시 앱은 "측정 중"에 무한히 머물렀습니다.

해결

sensors_attached 이후 600ms 지연 후 startCalibration 전송, 응답 미수신 시 25초 timeout으로 실패 상태 전환. 사용자가 무한 측정 화면에 갇히지 않게 됨.

session_result를 도메인 이벤트로 분리

문제

session_result payload는 운동 상태/세트 결과/반복/밸런스/근활성도 등 하위 구조가 복잡. 화면에서 바로 파싱하면 결과/기록/통계마다 다른 기준으로 값을 해석할 위험이 있었습니다.

해결

Pi session_result → DTO → WorkoutSession → LocalSessionRepository → SQLite 흐름 확립. workout_sessions_local 테이블 (15 cols / 3 indexes)에 payload_json + 요약 필드를 함께 저장해 업로드 실패 시 재시도 정책을 붙일 기반을 만들었습니다.

muscle_map 스케일과 key 불일치

문제

muscle_map이 0~1 ratio인지 0~100 percent인지 흔들리면 결과/기록/통계/히트맵이 서로 다른 숫자를 표시할 수 있었습니다. 일부 운동 mapping에서 센서 데이터가 빠지기도 했습니다.

해결

앱·Pi·백엔드 전 구간 0~100 percent로 통일(clamp 헬퍼 적용), 대표 region key 13개 + legacy alias 18개를 앱에서 정규화. 히트맵·리포트 숫자의 불일치 가능성을 줄였습니다.

CustomPainter 히트맵의 확장성 문제

문제

초기 히트맵은 painter 기반 도형에 가까워 실제 근육 부위 표현이 약했고, 성별/전후면/좌우/부위별 색상 변경을 확장하기 어려웠습니다.

해결

남/여 전·후면 SVG 4종 + flutter_svg ColorMapper로 SVG id별 색상 변경 구조. no data는 SVG 원본 neutral color, 측정된 0%는 별도 회색으로 분리.

키보드 overflow와 nested Scaffold

문제

로그인/회원가입/프로필 화면에서 키보드가 올라오면 overflow가 발생하고, 하단 탭 shell 안에 다시 Scaffold가 들어가면 앱바·바텀이 중첩되었습니다.

해결

LayoutBuilder + SingleChildScrollView + ConstrainedBox + IntrinsicHeight 패턴으로 키보드 대응. AppScaffold(nested: true)로 shell 내부 Scaffold 중첩 방지.

in the end

실시간을 안정적으로 다룬다는 것

가장 크게 배운 점은 실시간 센서 앱은 "데이터를 받는다"보다 "앱이 어떤 단위와 상태로 해석할지 정한다"가 더 중요하다는 점이었습니다. Pi에서 들어오는 메시지 빈도, 결측 상태, 센서 미부착 상태, 단위 스케일이 정리되지 않으면 화면은 있어도 사용자는 결과를 믿기 어렵습니다. 다음에는 시스템을 나누기 전에 먼저 데이터 계약과 테스트 케이스를 고정해두고 시작하고 싶습니다.
