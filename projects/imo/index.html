서비스 네이밍 & 캐릭터

IMO = Inside Muscle Out — 안에서부터 근육을 깨운다는 의미. 스마트글래스 캐릭터 아이(I)와 운동 캐릭터 모(Mo)로 브랜드를 구성했습니다.

내가 사용한 APP / FE 기술

실시간 센서 stream과 화면 상태의 균형을 잡기 위해, ViewModel / Repository / Service로 책임을 먼저 분리했습니다.

Provider / ChangeNotifier

Pi 실시간 stream을 UI 상태로 변환. notifyListeners() 호출 정책을 직접 제어하기 좋았습니다.

get_it (DI)

Service / Repository / UseCase / ViewModel 생성 책임 분리. 화면이 구현체를 직접 만들지 않도록 처리.

go_router

운동 준비 → 캘리브레이션 → HUD → 결과/기록/통계 등 긴 흐름을 명시적 route로 정리.

Dio Interceptor

access token 첨부, 401 refresh, 원 요청 재시도, 동시 401 single-flight 처리.

sqflite / SQLite

운동 결과를 서버 저장 전에 앱 내부에 먼저 저장(local-first). 네트워크 장애 시 결과 유실 방지.

flutter_svg

muscle_map key와 SVG path/group id를 연결해 부위별 색상 변경. CustomPainter 좌표 하드코딩보다 확장성↑.

트러블슈팅

실시간 센서·인증·도메인 계약·렌더링 안정성까지 앱 측에서 부딪힌 문제들과 해결.

인증

인증 토큰 갱신 중복 호출

문제

홈 진입처럼 여러 API가 동시에 호출되는 상황에서 access token이 만료되면 여러 요청이 동시에 401을 받아 각자 refresh를 호출했습니다.

원인

refresh token rotation 백엔드에서는 동시에 여러 번 refresh하면 둘째 요청부터 무효가 되어 세션이 통째로 날아갈 위험이 있었습니다.

해결

pendingRefresh Future를 공유하는 single-flight로 refresh API를 1회만 호출하고, 재시도 요청에는 authRetried 플래그를 달아 무한 재시도를 막았습니다.

배운 점

refresh 실패 원인을 unauthorized / network / server / unknown 4종으로 분리해, 진짜 무효(401·403)일 때만 logout하도록 정책을 나눠야 한다는 걸 배웠습니다.

✕ Before
onError: (error, handler) async {
  if (error.response?.statusCode == 401) {
    // 요청마다 각자 refresh 호출
    final newToken =
        await getIt<AuthRepository>().refreshToken();
    // 동시 401이면 refresh가 여러 번 → 둘째부터 invalid
  }
  handler.next(error);
}
✓ After
Future<String?>? pendingRefresh; // 공유

onError: (error, handler) async {
  final retried =
      error.requestOptions.extra['authRetried'] == true;
  if (error.response?.statusCode == 401 && !retried) {
    // single-flight: refresh API는 1회만
    pendingRefresh ??= getIt<AuthRepository>()
        .refreshToken()
        .whenComplete(() => pendingRefresh = null);
    final newToken = await pendingRefresh;
    if (newToken != null && newToken.isNotEmpty) {
      final req = error.requestOptions;
      req.extra['authRetried'] = true;
      req.headers['Authorization'] = 'Bearer $newToken';
      return handler.resolve(await dio.fetch(req));
    }
  }
  handler.next(error);
}
실시간

50Hz 표시 데이터로 인한 HUD 과다 재빌드

문제

glass_display_data가 약 50Hz로 들어오는데, 수신마다 notifyListeners()를 호출하면 HUD가 초당 50회 재빌드됐습니다.

원인

실시간 입력 빈도와 UI 표시 빈도를 동일하게 취급한 게 문제였습니다. 표시 데이터까지 매 프레임 갱신할 이유는 없었습니다.

해결

표시 데이터는 80ms throttle로 모아서 notify(50Hz → 약 12.5Hz, 호출 ~75%↓). 상태 이벤트는 즉시 반영해 실시간성과 안정성의 균형을 맞췄습니다.

배운 점

입력 빈도 ≠ 표시 빈도. 실시간 스트림은 수신과 렌더 갱신을 분리해서 다뤄야 한다는 걸 배웠습니다.

✕ Before
final snapshot = _snapshotAdapter.tryParse(message);
if (snapshot == null) return false;
_state = _snapshotMapper.map(snapshot);
if (notify) notifyListeners(); // 수신마다 → 50Hz 재빌드
✓ After
static const _interval = Duration(milliseconds: 80);
Timer? _timer;
bool _pending = false;

void _throttledNotify() {
  if (_timer != null) { _pending = true; return; }
  notifyListeners();
  _timer = Timer(_interval, () {
    _timer = null;
    if (_pending) { _pending = false; _throttledNotify(); }
  });
}

캘리브레이션 미응답과 Pi race condition

문제

sensors_attached 직후 start_calibration을 바로 보내면 Pi 상태머신이 phase 전환을 끝내기 전에 다음 명령을 받아 collection이 시작되지 않았고, 응답 미수신 시 앱은 "측정 중"에 무한히 머물렀습니다.

원인

앱이 Pi 상태머신의 phase 전환 시간을 보장하지 않았고, 응답 미수신을 실패로 판단할 기준도 없었습니다.

해결

sensors_attached 이후 600ms 지연 후 startCalibration 전송, 응답 미수신 시 25초 timeout으로 실패 상태 전환해 무한 측정 화면을 막았습니다.

배운 점

비동기 상태머신과 연동할 때는 전환 대기 시간과 타임아웃을 명시적으로 설계해야 한다는 걸 배웠습니다.

✓ After
Future<void> startCalibration() async {
  await Future<void>.delayed(
      const Duration(milliseconds: 600)); // Pi phase 전환 대기
  _send(StartCalibration());
  _timeout = Timer(const Duration(seconds: 25), () {
    if (state.isCalibrating) _markCalibrationFailed();
  });
}

  return _gradient(percent);                // 0~100%
}
